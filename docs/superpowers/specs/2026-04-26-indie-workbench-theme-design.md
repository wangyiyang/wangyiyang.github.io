# 独立开发者工作台主题重构设计

## 目的与结论

将博客从 fork 主题改造成一个克制、专业、长期耐看的独立开发者工作台。

新版主题不再把首页定义为传统文章列表，而是定义为个人生产系统入口：先表达一人公司方法论，再展示开源项目与工具能力，然后承接个人生产系统、专题路径和最新文章。

核心设计结论：

- 主题气质：极简技术杂志底色 + 独立开发者工作台结构。
- 首页顺序：个人品牌、专题体系、最新文章。
- 首屏定位：独立开发者 / 一人公司实践者。
- 首屏表达：方法论宣言型。
- 语气顺序：长期主义、观点鲜明、理性克制。
- 工作台模块顺序：开源项目 / 工具、个人生产系统、文章专题。
- 项目展示方式：能力模块分组、GitHub 数据辅助、精选代表项目控密度。
- 视觉密度：中等密度、信息清晰。
- 视觉语言：黑白灰 + 冷静蓝。

## 背景

当前博客基于 fork 主题继续演进，已有内容资产较丰富，但主题层仍保留明显的模板痕迹。继续修补零散问题可以改善局部体验，却无法解决以下核心矛盾：

- 首页缺少清晰的个人品牌叙事。
- 文章、专题、项目和开源之间没有形成统一入口。
- 旧主题结构让后续 SEO、搜索、性能和内容治理都只能以补丁方式推进。
- 构建和展示仍有 GitHub API 等外部依赖风险，不适合作为首页核心数据来源。

因此，本次优化应先做主题层重构，再把 SEO、搜索、构建性能和资源优化纳入新主题的工程收尾。

## 设计原则

### 第一性原则

这个博客的核心价值不是页面装饰，而是长期沉淀一套个人生产系统。主题需要服务三个目标：

- 让访问者快速理解作者是谁、关注什么、在构建什么。
- 让深度文章更容易阅读、检索和形成专题路径。
- 让开源项目、工具、文章和方法论互相支撑。

### 视觉原则

- 使用白 / 近白背景、黑色标题、深灰正文、浅灰分割线。
- 冷静蓝只用于链接、当前导航、CTA、项目元信息和重点状态。
- 不使用大面积渐变、装饰插画、卡片堆叠和强动画。
- 主要依靠排版、网格、留白、细线和信息层级建立品质感。
- 动效仅保留轻量 hover、focus 和必要的状态反馈。

### 工程原则

- 不修改 `_site/`。
- 不一次性删除旧主题，采用渐进式替换。
- 保持每个阶段可构建、可预览、可回滚。
- GitHub 数据不作为构建关键路径，优先使用 `_data` 静态数据或缓存。
- 遵循 DRY、KISS、SOLID、YAGNI，避免为主题重构引入过度抽象。

## 信息架构

### 顶部导航

导航保持轻量，但要从博客导航升级为产品入口：

- 首页
- 文章
- 专题
- 项目
- 开源
- 关于

移动端使用同样信息结构，收进简洁菜单，不引入复杂侧栏。

### 首页结构

首页采用中等信息密度，顺序如下：

1. 首屏方法论宣言
2. 工作台能力区
3. 代表项目 / 工具区
4. 个人生产系统区
5. 专题路径区
6. 最新文章区

### 首屏

首屏不是传统大 hero，也不是营销页，而是独立开发者工作台入口。

内容结构：

- 站点名：翊行代码
- 主标题：一人公司时代，开发者需要自己的 AI 工程系统。
- 副标题：我在这里记录如何用代码、AI、自动化和写作，构建个人生产基础设施。
- 主要入口：开源项目、生产系统、专题文章。
- 状态面板：AI Engineering / Automation / Writing / Open Source / Product Experiments。

视觉要求：

- 不使用大图、插画、渐变背景。
- 以大字号宣言、紧凑说明、细线网格和少量蓝色状态标识建立识别度。

### 工作台能力区

能力模块按以下方向组织：

- AI 工程
- 自动化工具
- 内容系统
- 产品实验
- 开源组件

每个模块展示一句定位和 1-2 个代表入口，避免首页变成仓库列表。

### 代表项目 / 工具区

项目展示以能力模块分组为主，GitHub 数据作为辅助，精选项目控制密度。

每个项目建议字段：

- 项目名
- 简短说明
- 技术标签
- GitHub 入口
- 相关文章入口
- stars、语言、更新时间等辅助信息

GitHub 数据应优先通过 `_data` 静态维护或脚本缓存生成，不依赖构建时实时请求。

### 个人生产系统区

用极简流程表达方法论闭环：

```text
输入信息 -> AI 辅助处理 -> 自动化执行 -> 产品/文章输出 -> 复盘沉淀
```

重点不是展示工具数量，而是说明 AI、代码、自动化、写作和产品实验如何组成个人生产基础设施。

### 专题路径区

专题不应只是分类聚合，而应像学习路径。首页优先展示：

- AI
- Agent
- RAG
- LangChain
- MLOps
- 架构
- 一人公司

每个专题展示文章数量、一句定位和入口。

### 最新文章区

最新文章放在首页后段，作为持续更新流。

展示字段：

- 标题
- 日期
- 分类
- 摘要

视觉保持列表化，不做卡片墙。

## 文章页设计

文章页是新版主题的核心体验，目标是长文阅读稳定、代码清晰、目录好用、移动端舒适。

### 文章头部

展示：

- 标题
- 日期
- 分类
- 阅读时间
- description 导语
- keywords / tags 辅助信息

keywords / tags 不抢标题层级。

### 正文区域

- 主栏宽度控制在适合长文阅读的范围。
- 重新定义字号、行高、段落间距、标题层级。
- 代码块、引用块、表格、图片和列表统一进入新主题样式。
- 代码块和链接状态使用冷静蓝作为有限强调。

### 目录

- 桌面端使用右侧 sticky TOC。
- 移动端使用文章头部下方折叠入口。
- 当前阅读位置使用冷静蓝标识。

### 文章尾部

包括：

- 上一篇 / 下一篇
- 相关推荐
- 回到专题入口
- 评论区

评论区保留，但视觉弱化，不压过正文。

### 特殊内容

继续兼容：

- Mermaid
- MathJax
- Mindmap
- 图片灯箱或宽图展示
- 视频嵌入

图片默认懒加载，视频默认轻量加载，避免拖慢页面。

## 二级页面设计

### 文章列表页

- 按时间流展示。
- 支持分类 / 标签筛选。
- 每篇展示标题、日期、摘要和分类。
- 使用列表 + 细线结构，不做卡片墙。

### 专题页

专题页应是一条阅读路径，而不是分类页换皮。

每个专题包含：

- 专题说明
- 推荐阅读顺序
- 核心文章
- 相关项目
- 延伸链接

### 项目页

项目页按能力模块分组，展示代表项目、工具、开源仓库和相关文章。

项目数据优先来自 `_data`，可以由脚本生成缓存，但页面渲染不能依赖实时 GitHub API。

### 关于页

关于页不写传统简历，而是围绕“如何构建个人生产系统”展开。

建议结构：

- 长期方向
- 工作方法
- 技术栈
- 代表内容
- 联系方式

## 组件体系

建议将主题组件拆为：

- `site-header`
- `site-footer`
- `hero-manifesto`
- `workbench-module`
- `project-list`
- `topic-path`
- `article-list`
- `post-header`
- `post-toc`
- `post-navigation`
- `tag-list`

CSS 建议按层组织：

- `tokens.css`：颜色、字号、间距、边框。
- `base.css`：正文、标题、链接、代码、表格。
- `layout.css`：页面网格、容器、响应式。
- `components.css`：组件样式。
- `pages.css`：页面级样式。

如仓库继续保留现有 CSS 文件，可以先通过新增主题入口文件聚合，再逐步清理旧样式，避免一次性大范围删除导致回归难以定位。

## 实施计划

### 阶段一：主题基础层

新增设计 token、基础排版、布局容器、组件样式。

目标：

- 视觉语言稳定。
- 不大改内容结构。
- 保持现有页面可构建。

### 阶段二：首页重构

将首页改为工作台结构。

包括：

- 方法论宣言
- 工作台能力区
- 代表项目 / 工具区
- 个人生产系统区
- 专题路径
- 最新文章

项目数据优先用 `_data` 静态维护。

### 阶段三：文章页重构

重做：

- 文章头部
- 正文排版
- TOC
- 上一篇 / 下一篇
- 相关推荐
- 评论区视觉层级

保留 Mermaid、MathJax、评论等现有能力。

### 阶段四：专题 / 项目 / 关于页

建设专题路径、项目页和新版关于页。

同步收敛分类与标签体系，减少一次性分类，细粒度主题放入 tags。

### 阶段五：旧主题清理与工程收尾

清理旧主题遗留：

- 不再使用的 CSS / include
- 本地资源路径问题
- Adsense 开关问题
- 搜索缓存策略
- GitHub API 构建依赖
- 图片 / 视频性能
- Front matter 内容质量检查

## 验收标准

每个阶段都需要满足：

- `bundle exec jekyll build` 可通过。
- 本地通过 `make serve` 或 `bundle exec jekyll serve` 预览真实 Jekyll 页面。
- 不修改 `_site/`。
- 移动端和桌面端主要页面无明显排版错位。
- 首页、文章页、专题页、项目页的视觉语言一致。
- GitHub 数据不可用时，核心页面仍可构建和阅读。

## 风险与应对

### 风险一：一次性改动过大

应对：按阶段拆 PR，先新增主题基础层，再迁移页面。

### 风险二：旧 CSS 和新 CSS 相互覆盖

应对：新主题入口使用明确命名空间或加载顺序，迁移完成后再清理旧样式。

### 风险三：GitHub API 影响构建稳定性

应对：核心项目数据改为 `_data` 静态数据或脚本缓存。

### 风险四：视觉改造影响阅读体验

应对：文章页作为核心验收对象，优先验证长文、代码块、目录、图片、移动端。

### 风险五：分类体系过散

应对：将主分类收敛为少数专题，细粒度关键词放入 tags。
