前阵子 GitHub 上爆火过一个项目，叫「三省六部·Edict」：作者把 1300 年前的唐朝官制搬进了多 Agent 架构——太子负责分拣消息，中书省规划，门下省审核封驳，尚书省派发，六部并行执行，还配了一块军机处看板。用户扮演皇上，说句话就是下旨，十几个 Agent 各司其职、相互制衡。项目上线两周星标 6.5K+，媒体给它起的外号叫「赛博朝廷」——顺便说一句，它的底座正是 OpenClaw。

大部分人围观的是文化梗：批奏折、下圣旨，好玩。但我盯着架构图看了很久，真正看进心里的，是作者写在 README 里的一句话：「古人比现代 AI 框架更懂分权制衡。」一千三百年前的人就想明白了：一个大脑管不住一个帝国，能管住帝国的是制度——职责有边界，决策有流程，执行有审计。

我立刻对号入座了。我没有帝国，只有一家一个人的公司；但我的烦恼，跟那位赛博皇上一模一样。

上周有个下午，我在四个小时里当了六种人：修联调 bug 的，是开发；按 1.91:1 模板出公众号封面、盯着朱红不许超过百分之四的，是设计；回读者留言的，是客服；核完订阅支出、顺手算了算跑道还剩几个月的，是财务；往稿件库拨进两个选题的，是编辑。最后一个动作，是打开 OKR 页问自己：这一下午，哪件事配写进周复盘？——第六种人上场了，CEO。

四个小时，六种人。没有谁在场超过一小时。

一人公司的真实死法，很少是累死的，多半是乱死的：每件事都在做，每件事都做了一半，复盘的时候连「我这周到底干了什么」都拼不出来。市面上的解法我基本都试过——GTD、PARA、时间块、番茄钟——它们都有用，但都没治到根上。直到我想通了那个赛博朝廷真正的启示：别人建的是朝廷，我该给自己注册一家「公司」。

先把结论摆这：**一人公司不是「我一个人扛」，而是「我一个人调度六个我」。** 调度系统，我用 Notion 数据库搭——说白了，就是给自己跑了一套 ERP。下面从「为什么拆部门」讲起，一路讲到一个看起来矛盾的结果：这套系统越建越大，我每天要面对的，反而不到三张表。

## 为什么是「部门」，不是「清单」

所有任务管理工具都在回答两个问题：事情放哪（PARA）、下一步做什么（GTD）。但一人公司真正的痛点是第三个问题：**这件事，现在该由「哪个我」来做？**

改 bug 的我和写文章的我，用的是两套完全不同的脑子。前者要收敛、要验证、要抠边界条件；后者要发散、要共情、要讲人话。这两个「我」来回横跳，切换成本比任何一件事本身都贵。

所以我把自己拆成了部门。每个部门有清晰的职责边界，更重要的是有清晰的**心智**——一句写在部门首页、每次进门都能看见的话：

- **CEO 办公室**：不做执行，只做方向选择 + 决策记录 + 周期复盘。「作为老板，千万不要太勤奋。」
- **编辑部**：低成本、轻流程、强闭环、能复盘。
- **设计部**：「一人公司干不起一件一设计，只能靠可复用的模板库。」这个部门有自己成文的 VI：墨黑打底、朱红点睛、一笔写就的 \{翊\}，从配色比例、字体到封面模板全部有规范可查——一个人的公司，也要长得像一家公司。
- **开发部**：「可交付 \> 完美，可重复 \> 一次性。」
- **增长部**：「一人公司最大的死因不是产品差，是没人知道。」
- **财务部**：「千万不要只有三个月的现金流，却做了三年产品。」这是六个部门里心态反差最大的一个：上班的时候，工资到账理所当然；自己坐上一号位，才第一次有了成本的概念——每笔订阅要抠，每个 Token 要算。我们不养闲人，也不养闲 Agent。

注意，这不是角色扮演的行为艺术。部门的本质是**给心智切换装一个开关**：打开编辑部的页面，我就只想选题和发布的事；打开财务部，我就只看收支和跑道。页面即语境，语境即角色。

![](/images/posts/yi-ren-gong-si-notion-erp/02.png)

这套开关最硬的证明，是坐进部门里的不一定是人。我给这家公司安排过一位 COO——一个 Agent：它读的是同样的表，认的是同样的状态流转，拆任务、派任务、盯进度，全靠任务卡异步接力，不需要我口头交代一句。每天早上我还没坐下，它已经把各部门的表扫了一遍，把当天的工作理好摆在桌上：今天该写什么、该修什么、什么在等我拍板——一号位的早晨从看简报开始，而不是从找今天该干嘛开始。

部门制走到这一步你才会发现，它真正定义的不是「我该怎么切换」，而是一个个**岗位**：职责、边界、流程全都写成了结构化的数据，坐上去的是人还是 Agent，公司都照转。

## 每个部门只留几张表

「用 Notion 搭系统」最大的坑，是系统本身变成新的负担——表建了二十张，维护成本比业务还重。我的原则反着来：**每个部门只留养得活的那几张表。**

拿编辑部举例，它只有两张表：

1. **稿件库**：从灵感到发布到复盘，一篇文章只占一行。看板就三列——选题池 / 撰写中 / 已发布。
2. **发布与数据**：每在一个渠道实际发布一次，补一行，记录阅读、点赞、收藏。

两张表之间用关联（Relation）打通，稿件库里的「总阅读量」「总点赞」全是 Rollup 自动汇总的。状态流转规则也定死了：拨到「撰写中」必须补立项日期和计划发布日；拨到「已发布」必须填实际发布日、同步在数据表补行。

就今天早上，我发了一篇文章。整个「行政流程」是：状态一拨、日期一填、数据表补两行，一分钟。下周五复盘的时候，这篇文章表现如何、跟同系列比怎么样，全都不用凭记忆——表里有。

稿件库还有一个上游：信息源。选题不能全指望灵感，灵感是个不稳定的供应商。所以我订了二十多个 RSS 源，从 InfoQ、Hacker News 到阿里技术，全部登记在一张「RSS 源管理」表里，每个源有权重、有状态；旁边配两张更小的表：正向关键词和反向关键词。过滤规则像海关——标题加摘要命中任一反向词，直接丢弃；命中正向词才放行，且反向优先于正向，宁可错杀，不能吵到我。于是每天流进选题池的不是整个互联网，而是机器初筛过一遍的候选。你看，连「注意力的入口」这件事，本身也是几张表。

其他部门同理：财务部就是收入台账 + 支出台账两张表，外加一个算跑道的公式；CEO 办公室是一个 PDCA 复盘库加一个 SOP 库；开发部干脆连表都少，因为代码的事实源在 GitHub，Notion 只留仓库索引和发版记录。

## 任务是消耗品，循环是复利

这些表里最值得单拎出来说的，是 CEO 办公室那个 PDCA 复盘库。我用过 Things，也用过飞书那套项目协同，它们都在帮你「把任务做完」——做完，这条记录的生命就结束了。而 PDCA 库管的是「循环转完留下什么」：每个日计划、周计划都是库里一行，Check 的时候挖根因，Adjust 把根因沉淀成 SOP；没解决的问题填进「遗留问题」属性，下一轮开 Plan 时第一个看它。任务清单是消耗品，循环是复利——同一个坑，在任务工具里你会掉进去三次，在 PDCA 库里它第二次就变成了标准。

PDCA 旁边还坐着一位书记官：ADR，决策记录。立库规则就一条——凡是选项超过一个、影响超过三个月的事，写一页。只记当时的处境和理由，不记马后炮；日后推翻了也不删，标成「已取代」、链向新的一条，每个选择的演化轨迹都留得住。复盘负责发现问题，ADR 负责锁定决策——一人公司没有同事帮你记得「当初为什么这么定」，这一页就是你和过去的自己的会议纪要。

更重要的是，这些方法论不是收藏夹里的干货文章，而是被直接缝合进了系统本身：Check 挖根因用的 5W2H 模板、Adjust 沉淀出的各种 SOP——写 Prompt 的、做用户访谈的、发文章的——全都住在 CEO 办公室的 SOP 库里，作为跨部门统一标准下发，哪个部门用到哪个环节，直接调用。方法论一旦上了表，就不再靠你「想起来」，而是流程走到那一步，它自己会出现。

**ERP 的核心从来不是软件，是「表 + 状态 + 流转规则」。** 表定义了什么信息值得留，状态定义了事情走到哪一步，流转规则定义了每一步必须交代什么。这三样有了，复盘就不再靠回忆，而是靠查询。

## 十几张表？我每天只开不到三张

看到这你可能要抬杠了：不是嫌系统重吗？稿件库、发布数据、信息源三件套、台账、PDCA、ADR、SOP——里里外外十几张表，比谁的系统都重。对，但这正是这套设计的全部要点：**表多，是系统的事；我每天要面对的，不到三张。** 早上看一眼简报，开一张当天的 PDCA 日计划；写作时段泡在稿件库；财务看板一周扫一次就够。

剩下那十几张表全在后台自转：Rollup 自动汇总数据，关键词海关自动初筛信息，SOP 等在流程该出现的那一步。而让它们转起来的，是 Notion AI——尤其是刚上的 Fable 5，专门为多步骤复杂 Agent 任务准备的那档模型：查表、汇总、补数据、生简报，这些「中层干部」的活现在都是它在干。制度负责定规矩，AI 负责跑规矩，我只负责拍板。

## 最狠的一招：部门是要解锁的

这套系统里我最得意的设计，是**部门不是一次建齐的，是打里程碑解锁的**：

- 产品部：全渠道粉丝破 1000 后解锁
- 商务部：获得首个付费客户后解锁
- 客服部：拥有 10 位真实用户后解锁
- 数据部：月收入突破 ¥3000 后解锁

为什么这么设计？因为一人公司最容易犯的错，就是**系统先于业务膨胀**——用户还没有一个，客服 FAQ 先写了三页；收入还没破千，数据看板先搭了五个。搭系统的快感是真的，但那是假装在工作。

解锁机制把这个冲动锁死了：部门页面就在那挂着，标题里明晃晃写着解锁条件。想开数据部？先去把月收入做到 ¥3000。系统反过来变成了业务的进度条。

## 为什么这比 GTD / PARA 更能落地

先说清楚：我没有抛弃 PARA，我的部门就住在 PARA 的 Areas 里。这三层各管一件事，谁也替代不了谁：

| 层 | 管什么 | 回答的问题 |
|---|---|---|
| GTD | 任务颗粒 | 下一步做什么？ |
| PARA | 信息归置 | 这个东西放哪？ |
| 部门 | 角色与流程 | 该由哪个我、按什么规矩做？ |

GTD 和 PARA 落不了地的原因，往往不是方法不对，而是它们**不带业务语义**：一条「写文章」的任务躺在清单里，和它的选题来源、目标渠道、发布数据毫无关系，做完就消失了，什么也没沉淀下来。而在部门制里，同一件事是稿件库里的一行，带着状态、日期和数据，做完之后它还在——变成了下一次决策的依据。

一句话：**GTD 管住了手，PARA 管住了柜子，部门制管住的是那个一天换六顶帽子的人。**

## 写在最后

有人可能觉得，一个人给自己开六个部门，有点装。我的看法恰好相反：正因为只有一个人，才更需要制度。真公司里角色切换有会议、有交接、有同事提醒你；一人公司里什么都没有，只有你和你的惯性。制度不是官僚，是给惯性修的护栏。

这套系统跑了一个多月，事情一件没少——六个部门的事照旧全在，只是它们各就各位了。我每天要面对的，从「所有的事」变成了不到三张表；最大的变化不是效率，是**心里有数**：每周五打开复盘页，这周发了什么、花了什么、什么在等我决策，全在表里。我不再需要「凭感觉」判断自己过得好不好。

> 一家公司里，一号位的时间最珍贵；而一人公司里，偷走一号位时间的，正是切换。部门不是装饰，是给每一次切换装上开关——你调度的不是任务，是六个不同的你。
