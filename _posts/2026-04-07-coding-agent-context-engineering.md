# 为什么同样的 Coding Agent，有人觉得神，有人觉得废

> 同一个 Coding Agent，在不同工程环境里会表现得像两个产品。差距越来越不只在模型，而在上下文工程。

很多人讨论 Coding Agent，会把体验差异归因于模型。

模型强，所以好用。
模型弱，所以不好用。

这个判断有一部分对，但不完整。

在真实项目里，同一个 Agent 放进两个环境，效果可能完全不同。

一个环境里，它能连续推进任务；另一个环境里，它频繁读错文件、误改模块、修一个问题带出三个问题。

差距往往不是模型突然变了，而是上下文系统不同。

Claude Code 官方文档里有一个很实用的提示：`CLAUDE.md` 会在会话开始时被读取，用来记录项目编码标准、架构决策、偏好库和 review checklist。这说明上下文工程不是玄学，而是 Agent 产品已经显式支持的工程接口。

## Agent 需要的不是更多信息，而是更对的信息

很多项目以为给 Agent 塞更多上下文就够了。

但上下文不是越多越好。

真正重要的是：

- 任务边界是否清楚；
- 仓库结构是否能快速理解；
- 本地规范是否写进 `AGENTS.md`；
- 测试命令是否明确；
- 哪些文件不能动是否提前说明；
- 失败时应该怎么接管。

这些信息缺失时，Agent 会自己猜。

Agent 一猜，风险就开始上升。

## 好的 Coding Agent 体验来自工作流，不只是模型

一个稳定的 AI Coding 工作流，至少包含四层：

- 任务描述：要解决什么，不解决什么；
- 上下文索引：关键目录、规范、依赖、入口；
- 验证反馈：测试、lint、构建、截图或运行结果；
- 人工接管：高风险改动和不确定判断必须停下来。

没有这四层，Agent 再强，也只能靠概率推进。

## 为什么有人觉得它"废"

很多失败体验，本质上是环境没有给 Agent 足够的边界。

比如让它"优化一下登录模块"，但没有说性能目标、兼容边界、测试范围、历史约束。

Agent 当然可以开始改。

但它很可能改到看似合理、实际不符合项目约束的方向。

用户看到结果，就会觉得它不靠谱。

## 为什么有人觉得它"神"

相反，如果项目有清晰规范、可运行测试、明确验收标准，Agent 的能力会被放大。

因为它每一步都能获得反馈。

它不是凭空"聪明"，而是被工作流约束得更稳。

## 先给结论

Coding Agent 的体验差异，越来越像工程系统差异。

模型能力是上限，上下文工程决定下限。

如果你希望 Agent 更像一个可靠同事，不要只换模型。先把任务边界、项目规范、验证命令和接管机制补齐。

真正让 Agent 变强的，不是无限上下文，而是可执行上下文。

参考资料：

- https://code.claude.com/docs/en/overview

## 一个很常见的对比

同样让 Agent 修一个接口 bug。

项目 A 只有一句需求：

> 登录接口偶尔失败，帮我修一下。

项目 B 提供了这些信息：

- 失败日志位置；
- 鉴权模块入口；
- 本地测试命令；
- 最近相关 PR；
- 不能改动的兼容逻辑；
- 修复后必须补的回归测试。

两边用的是同一个模型，结果会完全不同。

在项目 A 里，Agent 会先花大量时间猜：接口在哪、失败是什么、测试怎么跑、边界是什么。它越猜，越容易走偏。

在项目 B 里，Agent 更像一个接到清晰任务的工程师。它可以直接进入问题空间，把精力放在修复和验证上。

所以很多时候，所谓"Agent 好不好用"，其实是在评价项目有没有准备好让 Agent 工作。

## AGENTS.md 的价值会越来越高

很多团队低估了项目说明文档。

对人类新人来说，项目规范可以靠口口相传。对 Agent 来说，没有写下来的规则就等于不存在。

一个好的 `AGENTS.md` 不需要很长，但要回答关键问题：

- 这个仓库的主要目录是什么；
- 常用命令是什么；
- 新增代码放哪里；
- 什么风格不能破坏；
- 测试和验收怎么做；
- 高风险操作如何处理。

这份文档不是形式主义，而是给 Agent 的工作边界。

更进一步，官方文档还把 custom commands、hooks、skills 和 MCP 放在同一组扩展能力里。对团队来说，这意味着"让 Agent 好用"不是写一段更长 Prompt，而是把重复流程、外部工具和验证动作都接进工程系统。

## 人也要学会给 Agent 分任务

很多人第一次用 Coding Agent，会把任务扔得太大。

比如：

> 帮我重构一下整个支付模块。

这种任务对人类团队都需要设计评审，对 Agent 更不应该一次性放开。

更好的提法是：

1. 先让它阅读支付模块并总结边界；
2. 再让它列出重构风险；
3. 选一个低风险函数写测试；
4. 小步修改；
5. 跑测试；
6. 再决定下一步。

Agent 不是不能做复杂任务，而是复杂任务必须被拆成可验证的小步。

## 可执行上下文应该长什么样

如果你觉得某个 Coding Agent 很废，先不要急着换工具。

先检查你的项目是否给它提供了四样东西：

- 清晰任务；
- 可读上下文；
- 可执行验证；
- 可控接管。

缺了这四样，再强的 Agent 也只能靠猜。

一个真正对 Agent 友好的任务，通常会长这样：

1. 问题现象：用户登录后偶发 401；
2. 影响范围：只发生在刷新 token 后的部分请求；
3. 相关入口：`auth/middleware`、`tokenService`、`login.spec`；
4. 禁止事项：不要改数据库 schema，不要替换鉴权方案；
5. 验证命令：运行鉴权相关单测和登录端到端测试；
6. 交付要求：说明原因、改动点和剩余风险。

这不是在"喂答案"，而是在给 Agent 一个工程任务的基本工作条件。

## 最后：上下文工程决定 Agent 下限

模型能力决定上限，上下文工程决定下限。

一个没有规范、没有测试、没有边界的项目，会把强 Agent 变成随机改代码的工具。

一个结构清晰、反馈明确、允许小步验证的项目，会让同一个 Agent 表现得像可靠同事。

所以想让 Coding Agent 更好用，第一步不是换工具，而是把项目改造成 Agent 能工作的环境。
